Why AI Agents Need llm-d: The Inference Pattern Nobody Planned For

Markell Rawls, Developer Advocate, Red Hat
June 2025

I built an AI agent that worked beautifully in dev. It could research a topic, synthesize findings from multiple sources, answer follow-up questions, and even trigger downstream workflows. It felt like magic -- the kind of demo that makes a conference room go quiet. Then I deployed it to production, watched real users hit it, and saw the inference bill.

The cost was not what I expected. Neither was the latency. The numbers did not just creep up from my dev environment estimates -- they leaped. And as I dug in, I realized the problem was not the model, the prompts, or even the traffic volume. The problem was how the infrastructure was routing my agent's requests. Every single inference call was being treated like a stateless web request, scattered across GPU pods with zero awareness of what came before. My agent's carefully orchestrated reasoning chains were being shredded by a load balancer that had no idea what it was doing.

That experience sent me down a rabbit hole. What I found on the other end was llm-d -- and a fundamentally different way to think about inference routing for agentic workloads.

Agent Traffic Is Not API Traffic

If you have been building traditional API services, you have internalized a mental model: requests are mostly independent, roughly uniform in cost, and can be distributed across backends with simple strategies. Round-robin works. Least-connections works. These heuristics have powered the internet for decades.

AI agents break every one of those assumptions.

Multi-Turn Conversations

When a user has a five-turn conversation with an agent, that is not one inference call -- it is five, each one carrying the growing context of everything that came before. Turn one might be 500 tokens. Turn five might be 4,000. Each turn is a separate HTTP request, and each one benefits enormously from landing on a GPU pod that already has the previous turns' computed context in memory.

Tool-Calling Chains

Modern agents do not just generate text. They call tools. An agent handling a research question might call a search API, analyze the results, call a database, summarize its findings, then format a response. That is five or more inference calls from a single user request -- each building on the same base context. In the tool-calling pattern, the agent sends partial results back to the model repeatedly, and each round trip is a new inference request with a shared prefix.

Burst Patterns

Agents do not politely queue their requests and wait for responses. When an agent decides it needs to search three databases in parallel, it fires three inference calls simultaneously. When ten users trigger the same agent workflow in the same minute, you get burst patterns that look nothing like steady-state API traffic. The load is spiky, correlated, and unpredictable.

Context Switching

A single agent instance might be handling multiple users, each with different conversation histories, different cached contexts, and different topic domains. The infrastructure has to juggle all of these stateful sessions across a fleet of GPU pods, and getting it wrong means recomputing context from scratch -- the single most expensive operation in inference serving.

The Hidden Cost: KV Cache Misses

To understand why this matters, you need to understand one piece of GPU mechanics: the KV cache.

When a large language model processes your prompt, it computes attention keys and values for every token. These computed results -- the KV cache -- live in GPU memory. If your next request shares the same prefix (the same system prompt, the same conversation history, the same tool-call chain), the model can skip recomputing all of that and jump straight to the new tokens. This is the difference between a Time to First Token (TTFT) of 500 milliseconds or more and a TTFT under 100 milliseconds.

Now here is the problem: a standard Kubernetes load balancer -- round-robin, least-connections, random -- has no concept of KV cache state. It does not know which GPU pod has which context cached. It just distributes requests.

~5% Cache hit rate with round-robin routing (agent workloads)
80-95% Cache hit rate with llm-d prefix-aware routing

For agent workloads, round-robin routing achieves roughly a 5% KV cache hit rate. That means 95% of your inference requests are cold starts -- the model recomputes the entire context from scratch, burning GPU cycles and wall-clock time. Your users feel it. Your infrastructure budget feels it.

How llm-d Fixes This

llm-d is an inference gateway built for exactly this problem. It is not a model server -- it sits in front of your vLLM backends and makes routing decisions based on what is actually happening on the GPUs. Here is how it works.

Prefix-Aware Routing

When a request arrives, llm-d hashes the prompt prefix and matches it against a registry of which pods currently hold which cached prefixes. If pod 3 already has the first 2,000 tokens of this conversation cached in its KV store, the request goes to pod 3. Not randomly. Not round-robin. Intentionally, because that is where the work has already been done.

Cache-Aware Scheduling

The core intelligence lives in the EPP -- the Endpoint Picker Policy. The EPP maintains a real-time view of cache state across the entire fleet. It knows which pods hold which KV cache entries, how full each pod's cache is, and how recently each entry was accessed. This is not guesswork -- it is informed scheduling based on actual GPU memory state.

Load-Aware Balancing

Cache affinity alone would be dangerous. If one pod happens to cache a popular prefix, naive affinity would send all traffic there, overloading it while other pods sit idle. llm-d balances cache affinity against real-time load metrics. If the ideal pod is saturated, the request goes to the next-best option -- maybe a pod with a partial prefix match, or if necessary, a cold pod that can absorb the load. The system optimizes for throughput, not just cache hits.

The Results

2-5x TTFT improvement
40-60% Fewer cold cache starts

In practice, this translates to TTFT improvements of 2-5x and a 40-60% reduction in cold cache starts. For the end user talking to your agent, the difference is palpable -- responses that felt sluggish become snappy. For your infrastructure team, it means fewer GPU pods doing redundant work, which translates directly to cost savings.

Why This Belongs in the CNCF Ecosystem

llm-d is a CNCF Sandbox project, and that matters for reasons beyond prestige. It means the project is built to work with the tools platform teams already use.

The EPP runs as a Kubernetes-native component, integrating with the Gateway API -- the same standard that is replacing Ingress for HTTP routing. If your team already manages Kubernetes clusters and understands Gateway API resources, deploying llm-d does not require learning an entirely new operational model. It extends what you already know.

It works with any vLLM backend, which means you are not locked into a specific model serving stack. Swap models, scale pods, upgrade vLLM versions -- the routing layer adapts. This composability is exactly the kind of thing the CNCF ecosystem is designed to foster: focused components that do one thing well and integrate cleanly with the rest of the stack.

For platform teams that are being asked to support AI workloads alongside traditional services, this is significant. You do not need to build a separate operational universe for inference. You extend your existing Kubernetes platform with inference-aware routing, and your existing monitoring, scaling, and deployment pipelines continue to work.

The Bigger Picture

We are at an inflection point. The industry spent the last two years optimizing model training -- bigger clusters, faster interconnects, better parallelism strategies. That work matters. But the next wave of AI adoption is not about training. It is about inference at scale, and specifically about agentic inference, where the access patterns are fundamentally different from anything we have optimized for before.

Agents are not a niche use case. Every major cloud provider, every enterprise AI platform, and an enormous number of startups are building agent frameworks. The volume of inference traffic from agents is growing faster than the volume from direct human-to-model interactions. And all of that traffic has the same characteristics: multi-turn, tool-heavy, bursty, and deeply stateful.

The infrastructure layer has to catch up. Treating inference requests as stateless HTTP calls was fine when a user sent one question and got one answer. It falls apart when an agent sends thirty inference calls in ten seconds, each one depending on the context built by the ones before it.

llm-d is one of the first projects to take this problem seriously at the routing layer. It is not the only piece of the puzzle -- you still need efficient model servers, smart autoscaling, and thoughtful prompt engineering. But without cache-aware routing, you are leaving performance and money on the table at the layer where it matters most: the moment a request meets a GPU.
