Why Agentic Workloads Are the Killer App for KV Cache-Aware Routing

The era of single-shot LLM calls is ending. The pattern that dominated 2023 and early 2024—send a prompt, get a completion, done—is giving way to something far more demanding: agentic workloads. An AI agent does not make one LLM call per user request. It makes five. Ten. Sometimes thirty or more. Each call represents a step in a reasoning chain: interpreting the user's intent, deciding which tool to invoke, parsing the tool's output, deciding the next action, synthesizing a final response, and checking its own work along the way.

This is the fundamental shift that makes KV cache-aware routing not just useful, but essential. When you are running a simple chatbot, a cold cache miss on a single call adds maybe 400 milliseconds of extra prefill time. Annoying, but tolerable. When you are running an agent that chains eight LLM calls sequentially, that 400ms penalty compounds. You are not just adding latency to one response; you are adding it to every step of the agent's reasoning loop.

Consider a concrete example. A customer support agent receives a user message: "I need to return the shoes I ordered last week." The agent's execution looks something like this: (1) parse the user intent, (2) call a tool to look up recent orders, (3) interpret the order data, (4) call a tool to check the return policy, (5) interpret the policy, (6) call a tool to initiate the return, (7) confirm the return was successful, (8) compose a final response to the user. That is eight LLM calls, each of which includes the same system prompt and tool definitions as its prefix. Without cache-aware routing, every single one of those calls lands on a random pod and pays the full prefill cost for the entire shared prefix.

The math is straightforward and damning. If each call carries roughly 2,300 tokens of shared prefix (a typical system prompt plus tool definitions), and your average prefill throughput is around 5,000–6,000 tokens per second on a well-utilized GPU, you are spending approximately 400ms just recomputing the KV cache for that prefix on every call. Multiply by eight calls: that is 3.2 seconds of pure waste. The user is staring at a loading spinner for over three seconds longer than necessary, not because the model is slow, but because the infrastructure is doing redundant work.

Key Insight

The Compound Latency Effect

In traditional single-call LLM applications, a cache miss adds a few hundred milliseconds—barely noticeable. In agentic workloads, that penalty compounds across every step of the reasoning chain. Ten calls with 400ms of overhead each means 4 seconds of wasted compute. At scale, this is the difference between a snappy agent and one that feels broken.

This is precisely the problem that llm-d solves. llm-d is a CNCF Sandbox project purpose-built for distributed LLM inference routing. It sits in front of your vLLM backends and uses its EPP (Endpoint Picker Policy) component to make intelligent routing decisions. When an agent's first call lands on a pod and computes the KV cache for its prefix, llm-d ensures that all subsequent calls from the same agent are routed to that same pod. The prefix is already cached. Calls two through eight skip the prefill entirely for the shared tokens and go straight to computing attention only for the new, call-specific tokens.

The result is dramatic. Instead of 3.2 seconds of redundant prefill across eight calls, you pay 400ms once on the first call and roughly 65ms on each subsequent call (for the small amount of new, call-specific context). Total prefill time drops from 3.2 seconds to 855 milliseconds—a 73% reduction. That is not an optimization. That is a categorical change in the user experience.

The Prefix Cache Advantage: System Prompts and Tool Definitions as Shared Prefixes

To understand why llm-d's approach works so well for agentic workloads, you need to understand how KV caches work at the GPU level and why prefixes are the natural unit of caching.

When a transformer model processes a prompt, it computes key and value tensors for every token at every layer of attention. These tensors are the model's "memory" of the tokens it has seen—they encode the contextual relationships between tokens that the model needs to generate its output. For a model like Llama 3.1 70B with 80 attention layers and a hidden dimension of 8192, each token produces a KV pair that consumes a non-trivial amount of GPU memory and, critically, a non-trivial amount of compute to generate.

The key insight is that KV cache entries are deterministic for a given prefix. If two requests share the same first 2,000 tokens, their KV cache entries for those 2,000 tokens are byte-for-byte identical. There is no randomness in the attention computation during prefill—the model computes the exact same key and value tensors regardless of what comes after the prefix. This means that once you have computed the KV cache for a prefix on a specific GPU, any subsequent request that starts with that same prefix can reuse those cached tensors and skip directly to computing attention for the new tokens.

Why System Prompts and Tool Definitions Are Ideal for Caching

In agentic architectures, the shared prefix is not some small header or boilerplate. It is substantial, and it is the most computationally expensive part of the prompt to process because it comes first and establishes the context window for everything that follows.

A typical agent system prompt runs between 500 and 2,000 tokens. It defines the agent's persona, behavioral constraints, output formatting rules, and domain-specific instructions. This is not a one-line "You are a helpful assistant." It is a detailed specification that might look like: 

You are a senior customer support agent for Acme Corp.
You handle returns, exchanges, and order inquiries.

BEHAVIORAL RULES:
- Always verify the customer's identity before accessing order data
- Never disclose internal pricing or margin information
- Escalate to a human agent if the customer expresses frustration
  three or more times
- Always confirm actions before executing them
...
(continues for 800+ more tokens of guardrails, tone guidance,
 output format specifications, and escalation policies)

On top of that system prompt sit the tool definitions. These are the structured schemas that tell the model what tools it can invoke, what parameters each tool accepts, and what each tool returns. For an agent with five or six tools, these definitions easily add another 200 to 1,000 tokens. Each tool definition includes its name, description, parameter schemas with types and descriptions, required fields, and often example invocations or edge-case guidance.

Without llm-d, every call recomputes attention for all 2,300 prefix tokens. With llm-d, call 1 computes the prefix and caches it. Calls 2 through N skip directly to the call-specific context—typically just 50–200 tokens of new conversation or tool output. That is a 90%+ reduction in prefill work per call.

The savings are even more pronounced with larger models. Prefill compute scales linearly with both token count and model size. A 2,300-token prefix on a 70B-parameter model takes roughly 400ms to prefill. On a 405B-parameter model, that same prefix might take over a second. In agentic workloads on frontier models, the cost of redundant prefix computation is not just an annoyance—it is the dominant latency bottleneck.

llm-d's EPP component tracks which pods have computed KV caches for which prefixes. When a new inference request arrives, the EPP examines the request's prefix and checks whether any active pod already has that prefix cached. If so, it routes the request to that pod. If not, it selects a pod using load-aware heuristics and marks that pod as the new cache holder for that prefix. This lookup is fast—on the order of microseconds—because the EPP maintains an in-memory index of prefix hashes to pod assignments.

Multi-Turn Conversation Routing: Keeping Full Conversation Context Cached Across Turns

The prefix advantage extends far beyond the initial system prompt and tool definitions. In multi-turn conversations—which are the norm for agentic workloads—each successive turn adds to the prefix. Turn 1's prompt is the system prompt plus tool definitions plus the user's first message. Turn 2's prompt is all of that, plus the model's first response, plus the user's second message. By turn 5, the "prefix" that is shared with the previous turn might be 6,000 tokens or more.

This is where sticky routing becomes critical. llm-d supports session-aware routing that keeps an entire conversation pinned to the same pod across turns. The mechanics are straightforward: the first request in a conversation is routed to a pod based on prefix matching or load balancing. llm-d then associates the conversation's session identifier with that pod. All subsequent requests carrying the same session ID are routed to the same pod, where the growing KV cache from previous turns is already resident in GPU memory.

Compounding Returns

The Growing Prefix Effect

Each turn of a conversation increases the cached prefix. On turn 1, you cache 1,500 tokens (system prompt + tools). By turn 3, you might have 3,500 tokens cached. By turn 5, 6,000 tokens. The savings compound with each turn because you are skipping more prefill work each time. This means that the longer a conversation runs, the more valuable cache-aware routing becomes.

There is a subtlety here that is worth calling out. When a conversation's KV cache grows large enough, it may exceed what a single pod can keep resident in GPU memory alongside other active requests. llm-d handles this through intelligent cache eviction policies. If a conversation's cache is evicted due to memory pressure, llm-d can either re-route the conversation to a pod with more available cache memory or accept the one-time recomputation cost and re-cache on the same pod. The EPP's awareness of per-pod memory utilization ensures that these decisions are made with full knowledge of the cluster state, rather than blindly sticking to a pod that no longer holds the relevant cache.

Multi-Agent Cache Sharing: How Agents with the Same Tools Benefit from Shared Routing

So far we have been discussing cache reuse within a single agent's execution or conversation. But the real power of llm-d's approach emerges when you consider cache sharing across different agents.

In production platforms, it is common to have multiple types of agents that share overlapping tooling. A Research Agent, a Code Agent, and a Customer Support Agent might all have access to a web_search tool, a knowledge_base_query tool, and a send_notification tool. Their system prompts differ, but their tool definition blocks may be partially or fully identical. This creates a cross-agent caching opportunity that traditional load balancers completely miss.

llm-d's prefix matching is not all-or-nothing. It operates at a granular level. If two agents share the first 800 tokens of their prompts (identical tool definitions) but diverge after that (different system prompts), llm-d can still route them to the same pod to take advantage of that 800-token shared prefix. The KV cache for those 800 tokens is reused, and only the divergent portion needs to be computed fresh. This partial prefix matching is one of the more sophisticated aspects of llm-d's EPP implementation, and it is what makes multi-agent cache sharing practical.

Multi-Tenant Power

Shared Prefixes Across Customer Agents

Consider a SaaS platform that hosts AI agents for 50 different customers. Each customer has a unique system prompt, but all 50 agents use the same set of tools (CRM lookup, email send, calendar check, knowledge base search). The tool definitions alone might account for 900 tokens of shared prefix. llm-d can route requests from all 50 agents to a set of pods that keep those shared tool definitions cached, saving 900 tokens of prefill computation on every single call across all customers.

At 1,000 requests per second across all tenants, that is 900,000 tokens per second of redundant prefill eliminated. At typical GPU prefill throughput, that translates to freeing up the equivalent of 150+ GPU-seconds of compute every second—a substantial portion of the cluster's total capacity returned to productive use.

This multi-agent cache sharing is particularly valuable in platforms that implement a "tool marketplace" pattern, where agents can be composed from a library of standardized tools. The more standardization there is in tool definitions, the more cache sharing llm-d can exploit. This creates a virtuous cycle: standardizing on common tool schemas does not just improve developer experience—it directly reduces inference costs through better cache utilization.

There is an architectural implication here worth dwelling on. Traditional load balancers treat every request as independent and optimize for even distribution. llm-d inverts this: it deliberately creates affinity between requests that share prefixes, concentrating them on the same pods. This is counterintuitive from a load-balancing perspective, but it is the correct optimization for LLM inference because the compute savings from cache reuse far outweigh the marginal cost of slightly uneven load distribution. llm-d's EPP balances these concerns by factoring in both cache affinity and current pod load, routing to a cache-holding pod only if it has sufficient capacity to serve the request without excessive queuing delay.

MCP and llm-d: How Model Context Protocol Workloads Flow Through the Routing Layer

The Model Context Protocol (MCP) is rapidly becoming the standard interface for connecting LLM agents to external tools and data sources. MCP defines a structured format for exposing tools, resources, and prompts to LLM-based applications. This standardization has a profound and underappreciated implication for inference routing: MCP creates predictable, standardized prefix patterns that llm-d can exploit for cache optimization.

When an MCP-compliant agent starts up, it connects to one or more MCP servers and retrieves their tool manifests. These manifests are translated into the tool definition blocks that form part of the agent's prompt prefix. Because MCP defines a standard schema for tool definitions—including name, description, input schema with JSON Schema types, and annotations—the resulting token sequences are highly regular and predictable across different agents that connect to the same MCP servers.

MCP Creates a Natural Caching Hierarchy

Consider the layers of an MCP-enabled agent's prompt:

  1. Layer 1: MCP tool definitions — shared across all agents connecting to the same MCP servers. Highly cacheable.
  2. Layer 2: Agent system prompt — shared across all instances of the same agent type. Cacheable per agent type.
  3. Layer 3: Conversation history — unique per conversation session. Cacheable only with sticky routing.
  4. Layer 4: Current user message — unique per request. Never cached.

llm-d can exploit every layer of this hierarchy. Requests from different agent types that share MCP servers benefit from Layer 1 caching. Requests from the same agent type benefit from Layers 1 and 2. Requests within the same conversation benefit from Layers 1, 2, and 3. The deeper the shared prefix, the greater the cache savings.

◆ MCP Flow Through llm-d

Step 1: Agent connects to MCP servers (filesystem, database, web search) and retrieves tool manifests.
Step 2: Tool manifests are serialized into the prompt's tool definition block (~700 tokens).
Step 3: Agent's system prompt is prepended (~1,200 tokens).
Step 4: First inference request arrives at llm-d with full prefix (1,900 tokens).
Step 5: llm-d routes to a pod, which computes and caches the KV for all 1,900 tokens.
Step 6: Subsequent MCP tool calls from this agent (and others sharing the same MCP servers) route to the same pod, skipping up to 1,900 tokens of prefill.

The combination of MCP and llm-d creates a particularly powerful pattern for tool-call-heavy workflows. Consider an agent that executes a multi-step task involving five consecutive tool calls. Each tool call follows the same pattern: the agent sends a request to the LLM with the full context (system prompt + tool definitions + conversation so far + tool result), the LLM decides on the next tool to call, and the cycle repeats. Without cache-aware routing, each of those five requests would recompute the entire growing prefix from scratch. With llm-d's sticky routing, only the incrementally new tokens (the latest tool result and the new user context) require fresh prefill computation.

⚡

MCP + llm-d is the inference stack for the agentic era. If you are building agents without cache-aware routing, you are leaving 70% of your latency budget on the floor. Every MCP tool call that hits a cold cache is a tool call that makes your user wait for work the GPU has already done.

MCP also introduces a concept of prompt templates—standardized prompt structures that MCP servers can expose for common tasks. When multiple agents use the same MCP prompt template, the resulting token sequences share even more prefix overlap, further amplifying llm-d's cache hit rates. As MCP adoption grows and tool definitions become more standardized across the ecosystem, the cache efficiency of llm-d-routed inference clusters will improve correspondingly—a trend that makes the MCP+llm-d combination increasingly compelling over time.

There is a practical consideration here for platform architects. If you control the MCP servers your agents connect to, you can optimize for cache efficiency by ordering your tool definitions consistently. llm-d matches prefixes from the beginning of the prompt, so placing the most commonly shared tool definitions first in the serialization order maximizes the length of the shared prefix across agents. This is a small implementation detail that can have an outsized impact on cache hit rates in multi-agent deployments.

The Agentic Scaling Challenge: Why Agent Workloads Are Burstier and Harder to Predict

Agentic workloads do not just add more inference requests—they fundamentally change the shape of traffic. Understanding this is critical for anyone building infrastructure to serve AI agents at scale.

The Amplification Problem

In a traditional LLM application, one user request produces one inference call. The traffic pattern is relatively predictable: if you have 100 users making requests at an average rate of 2 requests per minute, you expect approximately 200 inference calls per minute. You can provision and autoscale for this with standard techniques.

Agents break this model entirely. One user request might produce 3 inference calls (a simple lookup) or 25 inference calls (a complex multi-step research task). The amplification factor is variable and data-dependent—it depends on the agent's reasoning, the complexity of the user's request, and the results of intermediate tool calls. You cannot predict it from the input alone. An agent that usually makes 5 calls might make 20 if a tool returns an error and the agent decides to try alternative approaches.

Burst Patterns in Agentic Workloads

The burstiness problem is compounded by the fact that agents often generate parallel bursts of inference requests. When a research agent decides to search five different sources simultaneously, it fires off five inference calls at once. When a code agent needs to analyze three different files before deciding on a fix, it may kick off three parallel analysis calls. These bursts are sudden, short-lived, and intense.

The burst pattern also exhibits temporal correlation. If your platform serves 100 concurrent users and each user triggers an agent, those 100 agents do not neatly stagger their inference calls. They tend to cluster: many agents will be in their "tool-calling" phase simultaneously, generating parallel bursts that stack on top of each other. The result is a traffic pattern that looks less like a steady stream and more like a series of sharp spikes separated by relative calm.

Traditional autoscaling cannot handle these bursts effectively. Kubernetes HPA (Horizontal Pod Autoscaler) typically operates on a 15–60 second observation window, and spinning up a new vLLM pod takes 30–120 seconds (model loading alone can take 30+ seconds on large models, and that is before the GPU memory is allocated and the model weights are transferred). By the time the autoscaler detects the burst and provisions new capacity, the burst is already over. The new pods sit idle, costing money, while the next burst hits the existing pods that are still recovering from the previous one.

How llm-d's Load-Aware Routing Helps Absorb Bursts

llm-d addresses the burst problem through load-aware routing that operates at the request level, not the pod level. When a burst of requests arrives, llm-d's EPP evaluates the current queue depth and active request count on every pod in real time. It routes requests to pods with available capacity, spreading the burst across the cluster without relying on autoscaling to provision new resources.

Crucially, llm-d's cache-awareness and load-awareness work together as a unified cost function rather than competing priorities. The EPP does not blindly route all requests from a cache-sharing prefix to the same pod if that pod is overloaded. It implements a dynamic cost model that balances the compute savings of cache reuse against the queuing delay of routing to a busy pod. If the cache-holding pod's queue depth exceeds a configurable threshold, the EPP will route the request to a less-loaded pod and accept the cache miss, knowing that the queuing delay would have exceeded the cache recomputation cost anyway.

Design Principle

Cache Affinity Is Not Stickiness at All Costs

A naive cache-aware router would always send requests to the pod with the cached prefix, even if that pod has 50 requests queued. llm-d is smarter: it models the trade-off between cache recomputation time (hundreds of milliseconds) and queuing delay (potentially seconds) and makes a dynamic routing decision for every request. When the cluster is lightly loaded, cache affinity dominates. When the cluster is heavily loaded, load balancing dominates. The EPP smoothly transitions between these modes based on real-time cluster state.

This dynamic balancing is especially important during burst scenarios. When five parallel agent calls arrive simultaneously, the first one or two might route to the cache-holding pod. But if calls three through five would queue behind them, llm-d routes them to other pods with available capacity. Those calls pay a higher prefill cost, but they start executing immediately instead of waiting in a queue. The net result is lower end-to-end latency for the agent's parallel burst, even though individual cache hit rates are lower.

llm-d also provides metrics and signals that external autoscalers can use to make better scaling decisions. By exposing per-pod queue depths, cache utilization ratios, and request arrival rates through its metrics endpoints, llm-d gives the autoscaler a richer picture of cluster state than simple CPU or memory utilization. A custom autoscaler that integrates with llm-d's metrics can anticipate capacity needs based on the number of active agent sessions and their historical amplification factors, rather than reacting to pod resource utilization after the fact. This predictive approach to scaling is far more effective for bursty agentic workloads than reactive HPA rules.

Production Patterns for Agentic Inference

Deploying llm-d for agentic workloads in production requires more than just enabling cache-aware routing. The following patterns represent hard-won lessons from operating agentic inference at scale, and they apply whether you are serving dozens or thousands of concurrent agents.

Connection Pooling for Agent-to-Inference Communication

AI agents are chatty. An agent making 8+ inference calls per execution should not be opening and closing HTTP connections for each call. Connection pooling between the agent runtime and the llm-d ingress is essential. Without it, TCP handshake and TLS negotiation overhead adds 5–20ms per call—a cost that compounds across the agent's call chain just like prefill latency does.

The recommended pattern is to maintain a pool of persistent HTTP/2 connections between each agent runtime pod and the llm-d routing layer. HTTP/2's multiplexing allows multiple inference requests to fly concurrently over the same connection, which is critical for agents that make parallel tool calls. Configure the pool size based on the maximum concurrent inference requests per agent runtime pod, typically 10–50 connections for a pod running 20–100 concurrent agent executions. Monitor connection utilization to right-size the pool: too few connections causes head-of-line blocking on parallel bursts; too many wastes kernel resources on idle sockets.

Request Coalescing: Batching Shared-Prefix Calls

When multiple agents share the same prefix and send inference requests at roughly the same time, llm-d can coalesce these requests into a single prefill computation. Instead of computing the KV cache for the shared prefix once per request, the router identifies requests with matching prefixes within a short time window (typically 10–50ms) and batches their prefill computation on the same pod. Each request still gets its own completion, but the shared prefix is computed only once and the resulting KV cache is shared across all requests in the batch.

Request coalescing is most effective in multi-tenant platforms where many agents share the same tool definitions. If 20 requests with the same 800-token tool definition prefix arrive within a 30ms window, coalescing reduces the prefill compute for those 20 requests by 95%: the prefix is computed once and the 20 unique suffixes are batched together for efficient GPU utilization. This is conceptually similar to how vLLM's continuous batching works at the generation level, extended to the routing and prefill layers by llm-d.

Priority Queues: Differentiating Interactive and Background Calls

Not all inference calls within an agent execution are equally urgent. The final response to the user is latency-sensitive—the user is actively waiting. But intermediate tool-call decisions, background reasoning steps, and self-verification checks are less time-critical. Production agentic systems should tag their inference requests with priority levels, and llm-d should route accordingly.

◆ Priority Queue Strategy

P0 (Critical): Final user-facing response generation. Route to lowest-queue-depth pod, even if it means a cache miss. Target TTFT: <100ms.
P1 (Interactive): Tool-call decision steps where the user is actively waiting. Prefer cache-aware routing but accept load-based fallback. Target TTFT: <200ms.
P2 (Background): Self-verification, logging summaries, pre-emptive caching. Route to cache-holding pod regardless of queue depth. Target TTFT: <2s acceptable.

llm-d's EPP can incorporate request priority into its routing cost function. High-priority requests weight queue depth more heavily in the cost model (because waiting is unacceptable), while low-priority requests weight cache affinity more heavily (because cache recomputation is wasteful when there is no user waiting). This differentiation allows the cluster to serve latency-sensitive interactive requests with minimal delay while still maximizing cache utilization for background work that can tolerate higher latency.

Circuit Breaking: Handling Cache Thrashing

Cache thrashing occurs when a pod's KV cache is constantly being evicted and recomputed due to memory pressure from too many concurrent sessions. This creates a pathological feedback loop: llm-d routes requests to the pod because it ostensibly "has" the cache, but the cache has actually been evicted by the time the request is processed, so the request pays the full prefill cost plus the queuing delay from being routed to a specific pod instead of the least-loaded one. The user gets the worst of both worlds: cache-miss latency with affinity-routing overhead.

The solution is circuit breaking at the cache level. llm-d monitors each pod's cache hit rate in real time. When a pod's cache hit rate drops below a configurable threshold (for example, below 40% over the last 60 seconds), llm-d's EPP marks that pod as "cache-degraded" and stops routing cache-affinity requests to it. Instead, it falls back to pure load-based routing for that pod until the cache hit rate recovers above a recovery threshold (for example, 60%). This prevents the pathological feedback loop and ensures that cache-aware routing only kicks in when it actually provides a measurable benefit.

Implementing this requires the vLLM backends to expose cache statistics to the EPP—specifically, the number of cache hits versus misses over a rolling window. llm-d's health-check protocol includes these metrics, allowing the EPP to make informed circuit-breaking decisions without adding latency to the critical request path.

Graceful Degradation: When All Pods Are Cold

There are scenarios where cache-aware routing provides no benefit: cluster cold starts after a deployment, scaling events that bring up new pods with empty caches, or traffic spikes that exceed the cluster's total cache capacity. In these cases, llm-d must degrade gracefully to pure load-based routing without introducing additional latency from cache-lookup overhead.

llm-d's EPP implements a fast-path check: if no pod in the cluster reports a cache entry matching the request's prefix hash, the EPP skips the cache-aware routing logic entirely and routes based on load metrics alone. This fast-path avoids the overhead of searching for a cache match that does not exist, keeping the "all-cold" routing performance equivalent to a standard load balancer. As pods warm up and cache entries accumulate, the EPP automatically transitions back to cache-aware routing without any manual intervention or configuration change.

The graceful degradation strategy also applies at the prefix level. If a new agent type is deployed with a novel system prompt that no pod has cached yet, llm-d treats its first request as a load-balanced cold start and begins building cache affinity from that point forward. Within seconds of the first request, the new prefix is cached and subsequent requests benefit from cache-aware routing. The transition is seamless, automatic, and invisible to the agent application. This "warm on first use" behavior means that deploying new agent types or updating system prompts does not require any cache pre-warming steps—llm-d handles it as part of normal request processing.

Conclusion: The Infrastructure Agentic AI Demands

Agentic workloads are not a future possibility; they are a present reality. Every major AI platform is shipping agent capabilities, and the inference traffic patterns are shifting from single-shot completions to multi-step, tool-augmented reasoning chains. This shift exposes a fundamental inefficiency in traditional inference routing: the repeated recomputation of shared prefixes across an agent's call chain.

llm-d addresses this inefficiency at the routing layer, where it belongs. By maintaining awareness of which pods have cached which prefixes, and by intelligently routing requests to maximize cache reuse while respecting load constraints, llm-d eliminates up to 75% of prefill latency in agentic workloads. The savings compound across multi-turn conversations, multiply across multi-agent platforms, and scale with the growing adoption of standardized tool protocols like MCP.

The numbers are clear: for an agent making 8 inference calls per execution, llm-d reduces total prefill time from 3.75 seconds to around 1.1 seconds. At 1,000 concurrent agents, that translates to nearly 44 GPUs' worth of freed compute per minute. For a cluster running 8x A100 GPUs at $2.21/hr each, the prefill savings from cache-aware routing save approximately $4,200/month at moderate agent traffic. As agent complexity grows—more tools, longer conversations, more sophisticated multi-step reasoning—the savings only increase.

The agentic era demands inference infrastructure that understands the structure of its workloads. Random routing treats every request as independent. Cache-aware routing recognizes that in agentic systems, requests are deeply correlated—sharing prefixes, sharing sessions, sharing tool definitions. llm-d is built from the ground up to exploit these correlations, and that is why agentic workloads are its killer app.
